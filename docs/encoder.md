# Encoder Task

## Ответственный

Богдан

## Цель

Реализовать слой кодирования, который разбивает строковый секрет на `shares` по схеме Shamir.

## Контракт

```python
split_secret(secret: str, threshold: int, total_shares: int) -> list[str]
```

## Что приходит на вход

- `secret: str`
- `threshold: int`
- `total_shares: int`

## Что должно уйти на выход

- `list[str]`
- длина списка строго равна `total_shares`
- каждый элемент списка является одним валидным `share`

## Формат share

```text
sss1:<threshold>:<prime_hex>:<byte_length_hex>:<x_hex>:<y_hex>
```

Этот формат является контрактом между core и backend. Менять его без согласования нельзя.

## Обязательные проверки

- `secret` не пустой
- `threshold >= 2`
- `total_shares >= 2`
- `threshold <= total_shares`

## Что входит в задачу

- преобразование строки в внутреннее числовое представление
- генерация коэффициентов полинома
- вычисление `shares`
- сериализация каждого `share` в строковый формат

## Что не входит в задачу

- HTTP
- DTO
- UI
- логирование запросов

## Критерий готовности

- функция возвращает корректное количество `shares`
- результат можно передать в decoder без дополнительных преобразований
- при невалидном входе выбрасывается понятная ошибка

## Интеграция с backend

Backend передает в encoder ровно три поля:

- `secret`
- `threshold`
- `total_shares`

Encoder возвращает только список `shares` и не знает ничего про `requestId`, JSON и HTTP-статусы.

## Полезная практика DNP

- `contract-first`: формат `share` фиксируется заранее
- `single responsibility`: encoder делает только разбиение секрета
